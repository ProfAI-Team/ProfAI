1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
ProfAI is a web platform that enables university students to analyze their
professors' exam styles. Students upload past exam files; the system produces
question-type distribution, topic-based analysis, and a difficulty score.
The platform answers a single, focused question: "How does this professor
ask questions?"

Core features:
  - Past exam file upload (PDF, JPG, PNG)
  - Question-type distribution (multiple choice, classic, true-false)
  - Topic-based distribution analysis
  - Difficulty score calculation
  - Professor rating (difficulty + fairness on a 1-5 scale)
  - Professor search + filter (name, department, university)
  - Personal dashboard (uploads + contribution statistics)
  - Responsive design (mobile-friendly)

4. UML CLASS DIAGRAM (MVP)
--------------------------------------------------------------------------------
The project's core domain model consists of 7 classes + 1 enum. See the
file class-diagram.png for the full diagram.

  - User              : Registered student on the platform
  - Professor         : The professor whose style is analyzed
  - Course            : A course owned by a Professor
  - Exam              : An uploaded exam file belonging to a Course
  - ExamAnalysis      : 1:1 analysis output for an Exam (question types +
                        topics + difficulty score)
  - ProfessorRating   : A rating a User gives to a Professor
  - ExamType (enum)   : MIDTERM | FINAL | MAKEUP

Relationships:
  Professor   1 --->  *   Course               (one professor, many courses)
  Course      1 --->  *   Exam                 (one course, many exams)
  Exam        1 --->  1   ExamAnalysis         (each exam has one analysis)
  Professor   1 --->  *   ProfessorRating      (ratings a professor receives)
  User        1 --->  *   ProfessorRating      (ratings a user submits)
  User        1 --->  *   Exam                 (exams a user uploads)
  Exam          --->      ExamType             (enum usage)


5. OBJECT-ORIENTED (OOP) PRINCIPLES
--------------------------------------------------------------------------------

5.1. ENCAPSULATION
-------------------
Every domain class guards its own data and exposes only meaningful methods.
Direct database access is isolated to the Prisma layer; controllers and the
UI never touch that layer directly.

Examples:
  - User.password is stored hashed; the login() flow verifies the password
    without exposing it to the outside.
    File: server/src/controllers/authController.ts
  - Exam.fileUrl is a value computed based on the local or R2 backend;
    this computation is encapsulated inside the upload() method.
    File: server/src/services/analysisService.ts
  - ExamAnalysis.analyze() produces the result without leaking the AI
    provider details to callers.
    File: server/src/services/analysisService.ts

5.2. INHERITANCE / INTERFACE IMPLEMENTATION
--------------------------------------------
In TypeScript, class hierarchies and interface implementation are two faces
of inheritance. The project favours interface contracts at the domain layer
over "extends" chains, aligning with the "composition over inheritance"
principle.

Examples:
  - The ExamType enum is used by the Exam class; behaviour specific to the
    exam type is dispatched polymorphically inside ExamAnalysis.
  - The service layer uses the Provider Pattern — shared interfaces:
      * StorageProvider  (put, publicUrl, delete, listOlderThan)
      * PaymentProvider  (init, parseWebhook, refund)
      * AIProvider       (generate, generateStructured)
    Concrete implementations conform to these interfaces; a factory
    function or a registry decides at runtime which one to use.
    (Details: section 6 — advanced architecture, provider pattern diagrams)

5.3. POLYMORPHISM
------------------
The same interface can be swapped with different implementations. The
service layer does not know which concrete implementation is active; it
only calls through the interface.

Examples:
  - getStorage() — returns an R2StorageProvider when the R2_BUCKET env var
    is set, otherwise a LocalStorageProvider. The upload controller only
    calls storage.put().
    File: server/src/lib/storage.ts
  - Professor.getAnalysisSummary() aggregates multiple underlying
    ExamAnalysis objects; different exam kinds (MIDTERM/FINAL/MAKEUP) flow
    through the same method polymorphically.
  - paymentService.init() delegates to the PaymentProvider interface's
    init() method. iyzico and (reserved) Stripe both flow through the
    same interface.
    File: server/src/services/payment/iyzicoProvider.ts

5.4. COMPOSITION
-----------------
Larger behaviours are built out of "has-a" relationships between classes.
ExamAnalysis belongs to an Exam, an Exam belongs to a Course, a Course
belongs to a Professor; each layer keeps the details of the one below it
to itself.


6. ADVANCED ARCHITECTURE (Additional Diagrams)
--------------------------------------------------------------------------------
Beyond the MVP, the project abstracts three critical subsystems behind the
Provider Pattern. These subsystems demonstrate OOP principles concretely on
production code:

6.1. Storage abstraction         — storage-diagram.png
     (server/src/lib/storage.ts)
     Interface: StorageProvider
     Implementations: LocalStorageProvider, R2StorageProvider
     Factory: getStorage()

6.2. Payment providers           — payment-diagram.png
     (server/src/services/payment/)
     Interface: PaymentProvider
     Implementations: IyzicoProvider, StripeProvider (reserved)
     HMAC signature verification is encapsulated inside the provider.

6.3. AI provider registry        — ai-provider-diagram.png
     (server/src/services/llm/)
     Interface: AIProvider
     Implementations: GeminiProvider, ClaudeProvider,
                      OpenAIRealtimeProvider (reserved)
     Facade: ProviderRegistry — supports automatic fallback via
     withFallback().

These three subsystems all follow the same recipe:
  1. A single interface (the contract) is defined.
  2. At least two concrete implementations exist (dev/local + prod).
  3. A factory / registry picks the right provider at runtime.
  4. Consumer services only know the interface.

The Liskov Substitution Principle (LSP) is strictly honoured: nowhere in
the code is there a branch like `if (provider === "r2") ...`.


7. CLEAN CODE AND MODULARITY
--------------------------------------------------------------------------------

7.1. Layered architecture
    Backend: route -> controller -> service -> prisma -> PostgreSQL
    Controllers hold no business logic; they only validate and serialize
    at the HTTP boundary. All business logic lives in the service layer.

7.2. Single responsibility
    Each service file covers a single domain (analysisService,
    authController, professorService, ...). Services can be tested
    independently of one another.

7.3. Feature flags and configuration
    Storage backend (R2_BUCKET), payment sandbox (PAYMENT_SANDBOX), AI
    tier (GEMINI_FREE_TIER) — every toggle is managed via env vars. The
    code runs unchanged across different environments.

7.4. Async/await and error boundaries
    All async code is written using async/await; Promise chains are not
    used. Controllers catch errors with try/catch and delegate to the
    global error middleware. Silent fail (swallowing exceptions) is
    forbidden.

7.5. Test infrastructure
    Vitest 4 with a 4-worker parallel test suite. Each worker runs
    against its own PostgreSQL schema. 319 unit + integration tests;
    the entire suite passes green in ~20 seconds.

7.6. CI/CD pipeline
    GitHub Actions runs three parallel stages: backend lint+build,
    frontend lint+build, Docker image build. It triggers automatically
    on every push and pull request.
